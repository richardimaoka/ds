Time Sync

  • Time Stamp Counter (TSC) is the most common clock source-- potentially unstable
  • The clocks on all nodes should be synchronized.
  • You can use NTP (Network Time Protocol) or other methods.

The vast majority of Cassandra instances run on x86 CPUs, where there are multiple clock sources available.

The fastest and most common is called "tsc" or Time Stamp Counter, which unfortunately isn’t that stable.

This is the reason why most x86 have so much clock drift

Since it’s important that the clocks on all nodes be synchronized, you should use NTP (Network Time Protocol) or other methods.

This is required because columns are only overwritten if the timestamp in the new version of the column is more recent than the existing column.

Tools

dstat

  • Shows all system resources instantly
dstat -lrvn 10
  • load average (-l), disk IOPS (-r), vmstats (-v), and network throughput (-n)

  • flip through my screens (ctrl-a n)
  • look for vertical consistency (or lack thereof) and outliers
  • memory should be around 95% in-use (most of it in the cache column)
  • CPUs should be < 1-2% of iowait and 2-15% system time
  • network throughput should mirror whatever the application is doing

Reading vmstats

  • see all the key metrics in one view
  • you can rely on pattern recognition
  • fir up a simple cassandra-stress load and watch change in metrics
  • going from left to right on a dstat -vn 10:

run/blocked/new tasks

  • How many processes or threads were running/blocked/created during the sample window.

memory

  • Used = heap + offheap + ~500MB
  • Buffers should be a couple hundred MB
  • Cache should account for almost all the remaining memory
  • Free memory ~ 200-300MB

swap

  • swap columns-- should always be zeros
  • any swap activity whatsoever should be eliminated

disk throughput

  • How many bytes are going in and out of the storage every second
  • Size the cluster/tuning to keep it 25% or so below peak load

interrupts (int) & context switches (ctx)

  • An interrupt occurs when a device needs the CPU to do something
  • Context switches occur when the kernel has to switch out a task on the CPU
  • ctx is almost always > interrupts
  • ctx should be 20-40% higher than interrupts

user/system/idle/wait/hiq/siq CPU time

  • User time is CPU time used to run userland processes
  • System time is time spent in kernel code
  • Idle time is exactly as it sounds
  • hiq/siq—​time spent processing hardware and soft interrupts

network send/recv

  • Shows the amount of data flowing over the network
  • shows how much data is flowing over CQL and storage protocols
  • difference between network / disk io / CPU usage gives idea of how efficiently a system is running

htop

  • Top provides an ongoing look at CPU processor activity in real time
  • Not only will it show you how CPU is used, but it also lists the processes consuming the most CPU first
  • but it is fairly limited and doesn’t look nice
  • htop has a nice default display that shows per-core load and all the threads
